<objective>
Extend the dead-stock parser and upload route to capture the optional FRED Stock Valuation "Cost Ex" column, persist per-unit cost into the existing `dead_stock.cost_ex DOUBLE PRECISION` column, and surface negative-cost data quality warnings in the upload response. This plan delivers COST-01 and lays the data foundation for COST-02 / COST-04 (delivered in plan 12-02).

Purpose: Without per-unit cost stored at upload time, the GET /api/dead-stock-summary endpoint (plan 12-02) has no source of dollar-value data. This is a strictly additive change — the `cost_ex` column already exists in the schema (verified: schema.sql line 47), and "Cost Ex" already appears in HEADER_ALIASES (verified: parser.ts line 39).

Output:
- DeadStockRow interface gains `costEx: number` (NaN when missing)
- parseDeadStockFile reads `colMap["Cost Ex"]` after a header-level `hasCostColumn` check
- upload.ts dead_stock INSERT extends with `cost_ex` column via UNNEST
- Negative cost rows produce `DataQualityWarning { field: "cost" }` and are stored as NULL
- Upload response gains `warnings: DataQualityWarning[]`
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Hono Worker (POST /api/upload) | Multipart file upload from authenticated user; file content is untrusted user-supplied bytes |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01 | Tampering | parseDeadStockFile (cost_ex parsing) | mitigate | parseFloat returns NaN for non-numeric input; NaN converted to NULL before INSERT — no string injection into numeric column. Negative values are emitted as DataQualityWarning AND converted to NULL (D-09), preventing intentional negative-cost manipulation from skewing aggregates downstream. |
| T-12-02 | Information Disclosure | dead_stock INSERT (cross-org) | mitigate | INSERT runs through withOrgContext which sets request.jwt.claims.org_id; RLS policy `org_isolation` on dead_stock (schema.sql line 101) enforces the policy at the DB layer. org_id is sourced from `c.get('orgId')` (verified Clerk JWT), never from request body. Pre-existing — no new exposure introduced by adding a column. |
| T-12-03 | Denial of Service | parseDeadStockFile (large CSV cost column) | mitigate | Existing 5 MB file size cap (upload.ts line 20, MAX_BYTES) checked BEFORE arrayBuffer() per Pitfall 3 — caps memory regardless of column count. Adding cost_ex parsing adds O(n) work over the same row count — no amplification. |
| T-12-04 | Tampering | DataQualityWarning response | accept | Warnings echo back the SKU from the user's own uploaded file. The user is the only consumer of their own warnings (org-scoped). No XSS surface — frontend renders as text, not HTML. Risk: low. |
| T-12-05 | Information Disclosure | DataQualityWarning leaking column-presence info | accept | Warnings only fire when negative cost values exist; absence of warnings does not reveal whether cost column was present or absent (that signal is delivered via dead-stock-summary.hasCostData in plan 12-02). No cross-org leakage possible — warnings are scoped to the same request. |
| T-12-06 | Repudiation | dead_stock writes (cost mutations) | accept | Existing pattern — DELETE+INSERT is the project's chosen replace-store-data semantic. uploaded_at timestamp captures when the latest version was written. No audit log requirement in v1 (REQUIREMENTS.md "Out of Scope" lists audit history as v2). |
