## v1 Requirements (carried from v1.0)

### Authentication & Tenancy

- [x] **AUTH-01**: User can create an account and sign in via Clerk (email + social) — ✓ v1.0
- [x] **AUTH-02**: Each pharmacy group has isolated data — all queries are scoped to org_id extracted from the verified Clerk JWT (never from request body) — ✓ v1.0
- [x] **AUTH-03**: User without an active Clerk organisation is blocked at middleware before any data operation — ✓ v1.0

### File Upload

- [x] **UPLOAD-01**: User can upload a FRED Office ROU report (CSV or XLSX: Item Code, Item Description, ROU, SOH) for a named store — ✓ v1.0
- [x] **UPLOAD-02**: User can upload a FRED Office dead stock report (CSV or XLSX: Item Code, Item Description, SOH) for a named store — ✓ v1.0
- [x] **UPLOAD-03**: Uploaded store data persists in NEON Postgres; user does not need to re-upload all stores to run a new match — ✓ v1.0
- [x] **UPLOAD-04**: User can see when each store's data was last uploaded and replace it individually — ✓ v1.0
- [x] **UPLOAD-05**: Parser handles FRED-specific CSV quirks — UTF-8 BOM stripping, CRLF line endings, blank title rows before the header row — ✓ v1.0
- [x] **UPLOAD-06**: Parser handles XLSX files via SheetJS (CDN tarball); enforces 5 MB per-file size cap — ✓ v1.0

### Matching Algorithm

- [x] **MATCH-01**: System identifies SKUs in a store's dead stock report that appear in other stores' ROU data with ROU > 0 — ✓ v1.0
- [x] **MATCH-02**: System applies sell-through filter — only matches destination stores where ROU ≥ SOH / 12 — ✓ v1.0
- [x] **MATCH-03**: User can set a months-cover target; maximum transfer quantity = (cover × destination ROU) − destination existing SOH; result is clamped to ≥ 0 — ✓ v1.0
- [x] **MATCH-04**: When destination store's existing SOH already exceeds months-cover target, that store is excluded from results — ✓ v1.0
- [x] **MATCH-05**: Results are sorted ranged-first, then by ROU descending within each group — ✓ v1.0 (Phase 7 gap closure)
- [x] **MATCH-06**: `is_ranged` parsing accepts all truthy variants: `checked`, `yes`, `true`, `1`, `y` (case-insensitive) — ✓ v1.0 (Phase 7 gap closure)
- [x] **MATCH-07**: NaN and missing ROU/cost values are explicitly handled rather than silently defaulted to 0 — ✓ v1.0

### Results & Export

- [x] **RESULTS-01**: Match results displayed in a virtualized table — ✓ v1.0
- [x] **RESULTS-02**: User can export match results as a PDF (client-side via @react-pdf/renderer) — ✓ v1.0

### Freemium & Billing

- [x] **BILLING-01**: Free tier allows 1 match run per calendar month per org; enforced via atomic Postgres counter — ✓ v1.0
- [x] **BILLING-02**: User can see how many match runs they have used this month and the monthly limit — ✓ v1.0
- [x] **BILLING-03**: When free limit is reached, user sees an upgrade prompt with a CTA linking to Stripe checkout — ✓ v1.0
- [x] **BILLING-04**: Stripe integration for paid plan — subscription creation, webhook handling for plan activation/cancellation, unlimited match runs on paid tier — ✓ v1.0

### Brand & UI

- [x] **BRAND-01**: UI implements PharmIQ brand guide — teal `#0F766E` primary, amber `#D97706` accent, navy `#0F172A` dark base, Space Grotesk (headings) + Inter (body) — ✓ v1.0
- [x] **BRAND-02**: Dark mode toggle — ✓ v1.0

### Logic Audit

- [x] **AUDIT-01**: Existing Django matching logic audited — ✓ v1.0
- [x] **AUDIT-02**: Ported TypeScript matching function has unit test coverage for all documented algorithm cases — ✓ v1.0

## v1.1 Requirements — Reporting & Tiered Billing

### Visualisation

- [x] **VIZ-01**: User sees a pie chart of dead stock units per store when dead stock data is uploaded — ✓ Phase 13 (VIZ-01 satisfied)
- [x] **VIZ-02**: After running a match, user sees a grouped bar chart: current vs projected dead stock units per store — ✓ Phase 13 (VIZ-02 satisfied)
- [x] **VIZ-03**: Post-match view shows a "Net units recovered" KPI card — ✓ Phase 13 (VIZ-03 satisfied)

### Cost & Dollar Reporting

- [x] **COST-01**: Dead stock upload accepts an optional "Cost Ex" column; absence detected at header level; does not cause upload errors — ✓ Phase 12 (COST-01 satisfied)
- [x] **COST-02**: When cost data is present, dead stock dollar value displayed per store — ✓ Phase 12 (COST-02 satisfied)
- [x] **COST-03**: User can enter a single org-level total SOH $ value; app displays dead stock as a % of total SOH with amber/red benchmark indicators — ✓ Phase 14 (COST-03 satisfied, VERIFICATION.md passed re-verified 2026-05-13)
- [x] **COST-04**: When cost column is absent, cost report panel shows instructional message — ✓ Phase 12+14 (COST-04 satisfied)
- [x] **COST-05**: When cost data is present after a match run, app shows a "Recoverable value" KPI — ✓ Phase 14 (COST-05 satisfied, VERIFICATION.md passed re-verified 2026-05-13)

### Billing

- [x] **BILLING-05**: Plan limits — Free: 1 match/mo, 3 stores; Pro: 10 matches/mo, 10 stores; Enterprise: unlimited — ✓ Phase 15 (UAT 9/9 ✓ 2026-05-13)
- [x] **BILLING-06**: All limits enforced server-side: atomic match counter + distinct-store count query — ✓ Phase 15
- [x] **BILLING-07**: Existing Free-tier users with >3 stores not retroactively blocked — ✓ Phase 15 (grace period: store cap applies to new match runs only)
- [x] **BILLING-08**: Stripe Checkout supports both Pro and Enterprise price IDs; Pro→Enterprise avoids duplicate subscription — ✓ Phase 15 (in-place subscription update)
- [x] **BILLING-09**: Synchronous checkout confirmation — fetches session status and writes plan_tier before rendering — ✓ Phase 15
- [x] **BILLING-10**: `customer.subscription.updated` webhook handler writes plan_tier; idempotent via stripe_event_id — ✓ Phase 15
- [x] **BILLING-11**: User can manage subscription via Stripe Customer Portal — ✓ Phase 15 (UAT Test 3: portal opened ✓)
- [x] **BILLING-12**: Billing page shows current plan, usage, stores vs limit, 3-tier comparison — ✓ Phase 15
