Add a try/catch wrapper around the GET /stores handler body (upload.ts line 186-227) so that database errors produce a JSON 500 response instead of an unhandled exception.

The POST /upload handler already has this pattern (lines 22-181 wrapped in try/catch returning `c.json({ error: '...' }, 500)`). Apply the same pattern to GET /stores.

Replace the GET /stores handler at lines 186-227 with:

```typescript
uploadRoute.get('/stores', async (c) => {
  try {
    const orgId = c.get('orgId');
    const dbUrl = c.env.DATABASE_URL;

    const stores = await withOrgContext<
      Array<{
        id: string;
        name: string;
        store_number: string | null;
        created_at: string;
        rou_uploaded_at: string | null;
        ds_uploaded_at: string | null;
      }>
    >(
      dbUrl,
      orgId,
      (tx) => tx`
        SELECT s.id, s.name, s.store_number,
               s.created_at,
               MAX(r.uploaded_at) AS rou_uploaded_at,
               MAX(d.uploaded_at) AS ds_uploaded_at
        FROM stores s
        LEFT JOIN rou_data r ON r.store_id = s.id
        LEFT JOIN dead_stock d ON d.store_id = s.id
        WHERE s.org_id = ${orgId}
        GROUP BY s.id, s.name, s.store_number, s.created_at
        ORDER BY s.name ASC
      `,
    );

    // camelCase JSON keys per project conventions
    return c.json({
      stores: stores.map((s) => ({
        id: s.id,
        name: s.name,
        storeNumber: s.store_number,
        createdAt: s.created_at,
        rouUploadedAt: s.rou_uploaded_at,
        dsUploadedAt: s.ds_uploaded_at,
      })),
    });
  } catch (err) {
    console.error('[stores] handler error:', err);
    return c.json({ error: 'Failed to load stores. Please try again.' }, 500);
  }
});
```

This is a defensive improvement flagged in VERIFICATION.md as a warning-level anti-pattern. The functional logic is unchanged — only the error handling wrapper is added.
