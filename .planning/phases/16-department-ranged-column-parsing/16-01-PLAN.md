<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| FRED file → parseDeadStockFile | Untrusted file content crosses into parser; department is a raw string cell value |
| parseDeadStockFile → upload.ts UNNEST | Parsed department string enters SQL parameter as typed array element |
| NEON dead_stock → match.ts | DB returns department TEXT (nullable); must be coerced before entering MatchResult |
| match.ts response → frontend | MatchResult.department is a user-controlled string rendered in JSX |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-01 | Tampering | upload.ts UNNEST INSERT | mitigate | Department value is passed as `${departments}::text[]` — NEON tagged template literal parameterises the array, preventing SQL injection. Same mechanism already used for skus, descriptions, costs. No hand-rolled string concatenation. |
| T-16-02 | Information Disclosure | match.ts dead_stock SELECT | accept | RLS policy `org_isolation ON dead_stock FOR ALL` automatically covers the new `department` column — no per-column policy change needed. withOrgContext sets request.jwt.claims before the query. Verified from schema.sql line 111. |
| T-16-03 | Tampering | parser.ts HEADER_ALIASES lookup | accept | Department alias matching is case-sensitive exact match via `aliases.includes(cell)` in buildColumnMap — no regex, no injection surface. Unrecognised column names are silently dropped. |
| T-16-04 | Information Disclosure | match.ts API response | accept | department is a free-text field from the pharmacy's own FRED export — no cross-org data. RLS ensures org isolation at DB layer. Null→"" coercion prevents null leaking to frontend. |
| T-16-05 | Tampering | Input Validation — department string length | accept | NEON TEXT column has no length limit. FRED department values are short strings (e.g. "Pharmacy", "PBS"). Upload is already gated at 5MB file size (upload.ts MAX_BYTES). No further length check needed for Phase 16. |
</threat_model>
